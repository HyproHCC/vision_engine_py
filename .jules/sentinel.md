## 2026-07-16 - [Protocol Image Path Extension Validation]
**Vulnerability:** The TCP server's `inspect` and `teach` commands accepted any arbitrary string as the `image_path` field. If directory traversal sequences (like `../../`) or sensitive system file paths were supplied, the server would check for file existence and attempt to read them using `np.fromfile`. This could leak file existence information to unauthorized clients (via differentiating between E_IMAGE_NOT_FOUND and E_IMAGE_LOAD_FAIL) and lead to potential denial-of-service or arbitrary file interaction attempts.
**Learning:** Prioritize strict protocol boundary validation for all file-related inputs. Standardizing accepted inputs at the TCP boundary (`parse_request`) prevents downstream components (like `load_gray` or `imdecode`) from interacting with non-image files.
**Prevention:** Always define an explicit allowlist of file extensions (such as `.png`, `.bmp`, `.jpg`, `.jpeg`, `.tif`, `.tiff`) and enforce it strictly at the entry point of command dispatching or request parsing before any file operation takes place.
